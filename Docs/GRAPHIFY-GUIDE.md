# graphify — гайд для команды AgOS

> Карта связей всего проекта (код + доки) с ответами на вопросы «что с чем связано»
> и «что сломается, если тронуть X». Этот документ — как этим пользоваться, что это
> даёт и как встроено в наш процесс.

---

## 1. Что это и зачем (за 30 секунд)

graphify строит **граф знаний** репозитория: 3000+ узлов (файлы, функции, сущности,
концепты из доков) и связи между ними. Лежит в репо как `graphify-out/graph.json`.

Зачем именно нам:

| Польза | Как помогает |
|--------|--------------|
| 🛡️ **Защита от регрессий** | Перед правкой видно «что зависит от этого узла» → не ломаем работающее (наша главная боль из CLAUDE.md, HS-1…HS-6) |
| 🧭 **Ориентация без чтения всего репо** | Вопрос к графу = ответ за секунды, в ~10× меньше токенов, чем grep по всему проекту |
| 🚀 **Онбординг** | Новый человек/агент сразу видит карту: что центральное, что с чем связано |
| 🔗 **Неочевидные связи** | Граф находит мосты между модулями, о которых не подумаешь спросить |

Граница ответственности: это **карта, а не истина**. Не заменяет тесты, ревью и
`cross_check.sh`. Связи с пометкой INFERRED — это догадки модели, их надо проверять.

## 2. Установка (один раз на компьютер, ~1 минута)

```bash
uv tool install graphifyy        # CLI; или: pipx install graphifyy
```

Всё. Карта уже в репозитории — **строить ничего не нужно**.

Если пользуешься **Claude Code** — просто перезапусти его. Он сам подхватит:
- MCP-сервер graphify (граф как нативный инструмент агента);
- хуки, которые напоминают спросить граф перед `grep` по исходникам.

Проверка, что работает:

```bash
graphify explain "useAuth"
```

## 3. Ежедневные команды (их всего 4)

```bash
graphify query "как работает регистрация эксперта"   # ответ из графа по теме
graphify explain "RationTab"                          # что за узел + ближайшие связи
graphify path "RationTab" "supabase"                  # кратчайший путь между двумя сущностями
graphify affected "useRpc"                            # ⭐ что сломается, если тронуть узел
```

| Команда | Когда применять |
|---------|-----------------|
| `query "вопрос"` | Не знаешь, где в коде что-то живёт. Старт любой задачи |
| `explain "узел"` | Нашёл функцию/компонент, хочешь понять его роль и окружение |
| `path "A" "B"` | Надо понять, как связаны две части системы |
| `affected "узел"` | **Перед правкой** — оценить радиус поражения |

Имя узла — это label из графа (`useAuth`, `cn`, `ProjectWizard`, `RationTab`,
`run_calculation`). Если не уверен в названии — сначала `query`, он подскажет узлы.

В **Claude Code** то же самое доступно через `/graphify query "..."` и через
MCP-инструменты (`query_graph`, `get_neighbors`, `shortest_path`, `god_nodes`) —
агент вызывает их сам.

## 4. Как это встроено в наш процесс

Наш git-флоу (ветки, rebase, PR) — без изменений, см. [CONTRIBUTING.md](../CONTRIBUTING.md).
graphify встраивается в 4 точках:

**1. Перед правкой — оцени радиус (ручное, 5 секунд).**
```bash
graphify affected "<узел, который собираешься менять>"
```
God-узлы (самые связанные хабы) трогать = трогать пол-системы. Текущий топ:
`App.tsx`, `useAuth()`, `supabase`, `cn()`, `useSetTopbar()`, `useRpc()`.

**2. Во время работы — спрашивай граф вместо слепого grep.**
`query`/`explain` дают точечный подграф вместо простыни результатов.

**3. На каждый PR — бот комментит автоматически (ничего делать не надо).**
GitHub Action `graphify blast radius` оставляет на PR коммент: какие god-узлы
задеты и с какими файлами связаны изменения. Читай его как чек-лист для ревью.

**4. Раз в неделю — карта обновляется сама.**
Action `graphify weekly refresh` (пн 06:00 UTC) пересобирает AST-слой и открывает
PR `chore(graphify): weekly AST refresh`. **Смержи его в один клик** — и карта снова
актуальна. Если изменений нет, PR не создаётся.

## 5. Что покажет PR-бот и как читать

Под каждым PR появится коммент `🗺️ graphify — карта влияния`:

- **⚠️ Затронуты god nodes** — изменения задели хаб. Ревьюй особенно внимательно,
  выше риск зацепить смежное.
- **Изменённые файлы → что связано** — список файлов, на которые изменения могут
  повлиять через граф связей. Глянь, не забыл ли проверить смежное.
- **🆕 Новые файлы** — их ещё нет в графе, появятся после еженедельного рефреша.

Это подсказка по риску, **не** блокер мержа.

## 6. Кто и когда пересобирает граф

| Случай | Действие | Кто |
|--------|----------|-----|
| Обычная неделя | ничего — еженедельный CI сам обновит AST | авто |
| Крупный рефакторинг с удалением кода | `graphify update . --force` локально → закоммить graph.json в PR | автор рефактора |
| Значимые изменения в доках (Dok*, микрошаги) | `/graphify .` в Claude Code (полная пересборка с семантикой, нужны субагенты/Gemini-ключ) | один человек |

⚠️ Не ставь авто-хук пересборки на каждый commit (`graphify hook install`) — иначе
карта (несколько MB) будет лететь в diff постоянно. Пересборка — осознанная.

## 7. Если что-то не так

| Симптом | Решение |
|---------|---------|
| `graphify: command not found` | `uv tool install graphifyy` (или `pipx install graphifyy`) |
| MCP graphify не виден в Claude Code | перезапусти Claude Code; проверь, что `uv` стоит |
| PR-бот не прокомментировал | PR из форка (там токен read-only) — норм; для веток внутри репо работает |
| Граф кажется устаревшим | дождись еженедельного PR или `graphify update .` локально |
| Хук «MANDATORY: run graphify» мешает | это лишь подсказка; отключить — `graphify claude uninstall` |

---

## 8. TL;DR

- Поставь раз: `uv tool install graphifyy`.
- Перед правкой: `graphify affected "<узел>"`.
- Не знаешь где что: `graphify query "..."`.
- PR-бот и еженедельный рефреш — автоматические, от тебя только **смержить** недельный PR.
- Это карта и защита от регрессий, **не** замена тестам и ревью.
