# Git-процесс для команды AgOS

> Памятка по совместной работе двух разработчиков с разных компьютеров.
> Цель: каждый ведёт работу в своей ветке, изменения вливаются в `main` **поверх**
> актуального состояния, ничего не теряется.

---

## Главное правило

**Никогда не вливай в `main`, не подтянув сначала свежий `main` к себе в ветку.**

Конфликты разрешаются локально (где не страшно сломать), а в общий `main`
приходит уже готовый, совместимый результат.

---

## Почему чужие изменения НЕ теряются

Когда ты вливаешь свою ветку в `main`, git делает **слияние (merge)**, а не замену.
То, что уже лежит в `main` (изменения второго разработчика), никуда не девается.
Git берёт общую базу, твои изменения и его изменения — и складывает оба набора.

Опасность ровно одна: если вы **оба правили одни и те же строки одного файла** →
git не знает, чью версию брать, и просит разрешить **конфликт** вручную.
Разные файлы или разные места — сольётся автоматически, без потерь.

## Процесс по шагам

### 1. Начало новой задачи — стартуй от свежего `main`

```bash
git checkout main
git pull origin main                  # забрал последнее состояние main с GitHub
git checkout -b feat/<короткое-имя>    # своя ветка (см. префиксы ниже)
```

### 2. Работаешь и коммитишь по ходу

```bash
git add .
git commit -m "feat(registration): добавил intake формы"
```

### 3. ПЕРЕД вливанием — обнови `main` и наложи свою работу поверх

```bash
git checkout main
git pull origin main                  # тут уже могут быть изменения второго разработчика
git checkout feat/<твоя-ветка>
git rebase main                        # переставляет твои коммиты ПОВЕРХ обновлённого main
```

Если возник конфликт — git остановится:

```bash
# открой файл, реши какой код оставить (или оба куска),
# затем:
git add <файл>
git rebase --continue
# если запутался и хочешь отменить rebase целиком:  git rebase --abort
```

### 4. Отправь ветку и открой Pull Request

```bash
git push --force-with-lease            # после rebase нужен force (безопасный вариант)
```

Дальше на GitHub: **Pull Request → Merge в `main`**.

Второй разработчик перед своим вливанием делает **то же самое**: подтягивает свежий
`main` (уже с твоими изменениями), накладывает свою работу поверх, разрешает
конфликты у себя, и только потом мёржит.

---

## rebase или merge

Два способа «подтянуть main в свою ветку» на шаге 3:

| Способ | Команда | Что делает | push |
|--------|---------|-----------|------|
| **Rebase** (рекомендуем) | `git rebase main` | Переставляет твои коммиты поверх main → история прямая и чистая | `git push --force-with-lease` |
| **Merge** (проще) | `git merge main` | Вливает main в твою ветку отдельным merge-коммитом | обычный `git push` |

Рекомендуем **rebase** — линейная история, ровно картинка «мои коммиты лежат поверх
актуального main». Если `--force-with-lease` пока пугает — используй `git merge main`,
он безопаснее и прощает ошибки, просто история будет с лишними merge-коммитами.

> ⚠️ `--force-with-lease`, а **не** `--force`. Lease-вариант откажется пушить, если
> кто-то успел изменить ветку на сервере — защита от случайной перезаписи чужой работы.
> Force-пушить можно **только свою личную ветку**, никогда не `main`.

## Именование веток

Префикс по типу работы (как уже принято в репозитории):

| Префикс | Для чего | Пример |
|---------|----------|--------|
| `feat/` | новая функциональность | `feat/registration` |
| `fix/`  | исправление бага | `fix/vet-f11-isolation` |
| `chore/`| рутина, конфиги, зависимости | `chore/env-config` |
| `docs/` | документация | `docs/reconciliation-phase1` |
| `hotfix/`| срочное исправление в проде | `hotfix/vet-f11-isolation` |

---

## Шпаргалка «застрял»

| Ситуация | Команда |
|----------|---------|
| Посмотреть, что изменилось | `git status` / `git diff` |
| Узнать, отстала ли ветка от main | `git fetch && git log --oneline main..HEAD` |
| Отменить незакоммиченные правки в файле | `git checkout -- <файл>` |
| Прервать запутавшийся rebase | `git rebase --abort` |
| Прервать запутавшийся merge | `git merge --abort` |
| Случайно начал работу прямо в `main` | `git stash` → `git checkout -b feat/...` → `git stash pop` |

## Граф знаний (graphify)

В репозитории лежит **карта кодовой базы** — граф связей между файлами, доками и
сущностями. Она помогает быстро понять «что с чем связано» и «что сломается, если
тронуть X», не перечитывая весь репозиторий.

В git едут только `graphify-out/graph.json` (карта) и `graphify-out/GRAPH_REPORT.md`
(отчёт: god-узлы, сообщества, мосты). Тяжёлый `graph.html` и кэш — игнорируются.

### Настройка (один раз на компьютер)

```bash
uv tool install graphifyy        # или: pipx install graphifyy
```

После этого карта из репозитория сразу доступна — пересобирать ничего не нужно.

### Как пользоваться

```bash
graphify query "как работает авторизация"      # ответ из графа (дёшево, без чтения всего репо)
graphify explain "useRpc"                       # что это за узел и с чем связан
graphify path "RationTab" "supabase"            # кратчайший путь между двумя сущностями
graphify affected "useAuth"                      # что сломается, если тронуть узел (blast radius)
```

Пользователям **Claude Code**: команда `/graphify query "..."` и хуки в
`.claude/settings.json` подключаются автоматически.

### Пересборка (осознанно, не на каждый commit)

Граф — снимок на момент пересборки, поэтому со временем устаревает. Обновлять:

```bash
graphify update .     # быстрая пересборка по коду (AST, без LLM-затрат)
```

Делает это **один человек** (или CI) раз в спринт / после крупных изменений, затем
коммитит обновлённый `graphify-out/graph.json` отдельным `chore:`-коммитом. Авто-хук
пересборки на каждый commit намеренно **не ставим** — иначе карта (несколько MB)
будет попадать в diff постоянно. Полная пересборка с семантикой по докам — через
`/graphify .` в Claude Code (требует субагентов / Gemini-ключа).

### Автоматизация (GitHub Actions)

- **`graphify blast radius`** (`.github/workflows/graphify-pr.yml`) — на каждый PR
  комментирует, с чем связаны изменённые файлы, и флагает задетые god-узлы (хабы с
  высокой связностью). Это подсказка по риску регрессий, **не** замена ревью/тестам.
  Граф читается готовый из репо, ничего не пересобирается.
- **`graphify weekly refresh`** (`.github/workflows/graphify-refresh.yml`) — раз в
  неделю (пн 06:00 UTC) пересобирает AST-слой и открывает **PR** со свежим
  `graph.json` (мерж в 1 клик, 0 апрувов). Запуск вручную: вкладка
  **Actions → graphify weekly refresh → Run workflow**.
  ⚠️ Одноразовая настройка репозитория: **Settings → Actions → General →
  Workflow permissions** → «Read and write permissions» + галочка «Allow GitHub
  Actions to create and approve pull requests». Без этого шаг создания PR падает.

### MCP-сервер (нативный инструмент для агентов)

`.mcp.json` поднимает graphify как MCP-сервер для Claude Code — агенты запрашивают
граф напрямую (`query_graph`, `get_neighbors`, `shortest_path`, `god_nodes`…), а не
через bash. Требует `uv` на машине (`uvx` запускает сервер из пакета `graphifyy`).
